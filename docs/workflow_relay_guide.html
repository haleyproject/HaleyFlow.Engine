Protocol Runtime

WorkflowRelay is the zero-infrastructure runner for HaleyFlowProtocol.

It loads the same definition and policy JSON used by the engine, interprets the sequence locally, and advances one step at a time inside your process. It is meant for local orchestration, lightweight apps, testing, and early-stage workflows where you want the protocol semantics without standing up the full engine + consumer delivery pipeline.

What it gives you

Transition routing, gate/effect hook ordering, policy complete codes, parameter resolution, and inline progression with no external services.

What it does not do

No retries, no crash recovery, no timeout monitor, no ACK tracking, and no durable inbox/outbox semantics.

Best fit

Development, prototypes, internal tools, and apps that want protocol-driven workflow logic but do not need distributed delivery guarantees yet.

Execution

Execution model

WorkflowRelay.FromJson(definitionJson, policyJson) parses the protocol into a WorkflowDefinitionSnapshot, validates it, and then drives everything from NextAsync. The relay keeps sequence knowledge inside the runtime so your business handlers only return true or false.

StageWhat relay does
LoadParses states, transitions, policy rules, hook routes, complete codes, and parameter references from protocol JSON.
ValidateRuns PolicyValidator. Critical findings stop startup. Warnings are surfaced but do not block registration.
NextAsyncFinds the matching transition from ctx.CurrentState + eventCode, runs the transition handler, advances state, then runs hooks in order.
Auto-advanceUses policy complete.success / complete.failure or code overrides to decide the next event code inline.
Important: relay does not use consumer delivery concepts such as DispatchMode or the engine's separate Complete handoff. Relay is a single in-process interpreter, so it runs the whole sequence directly inside NextAsync.
Hooks

Gate and effect behavior

Relay follows the protocol hook ordering rules exactly:

Hook typeBehavior in relay
gate Runs in order. Failure stops the sequence immediately. If the gate has complete.failure, relay jumps to that event. If the gate succeeds with complete.success, relay remembers that code, skips remaining gates, still drains effect hooks, then fires the success event.
effect Runs as fire-and-continue work. Relay does not block progression on the returned boolean. It keeps moving after the effect handler finishes.
Missing order Defaults to int.MaxValue, so unordered hooks run last.
Tradeoff: because relay has no durable ACK pipeline, application code is responsible for idempotency. If a process crashes halfway through local work, relay will not replay from a stored inbox or outbox row.
Routing

Three routing tiers

Both transition handlers and hook handlers support the same precedence model:

TierMeaning
Tier 1Policy JSON complete codes are the default routing source.
Tier 2Code override on On(..., successCode, failureCode) or OnHook(...) when policy omits complete codes.
Tier 3A callback Func<bool, RelayContext, Task<int?>> decides the next event dynamically for advanced logic.

RelayContext carries the local entity reference, current state, actor, payload, and resolved parameter catalog entries for the transition or hook currently executing.

var relay = WorkflowRelay.FromJson(definitionJson, policyJson)
    .On(4000, async ctx => { await review.ValidateAsync(ctx.EntityRef); return true; })
    .OnHook("APP.REVIEW.AUDIT", async ctx => { await audit.WriteAsync(ctx.EntityRef); return true; })
    .SetMonitor((eventCode, entityRef) => monitor.AllowAsync(eventCode, entityRef));
Host

WorkflowRelayBase and WorkflowRelayHost

If you want relay execution managed by DI, use WorkflowRelayBase plus WorkflowRelayHost.

  1. Subclass WorkflowRelayBase.
  2. Decorate the class with [LifeCycleDefinition("definition-name")].
  3. Return protocol JSON from DefinitionJson and optionally PolicyJson.
  4. Implement ConfigureRelay(WorkflowRelay relay) and register On / OnHook handlers.
  5. WorkflowRelayHost discovers the relay at startup and routes FlowInitiateRequest by WorkflowName.
Host contract: FlowInitiateRequest.StartEvent must parse to an integer event code. Relay host converts it and then calls Relay.NextAsync(...).
Human wait/resume: WorkflowRelayHost can resume from the last saved state through IWorkflowRelayStateStore. The default store is in-memory for local execution; applications that pause for days or weeks should replace it with a database-backed implementation. Controller-level human actions should call IFlowBus.InitiateAsync with the completion event, while business services remain unaware of relay state. 
services.AddSingleton<IWorkflowRelayStateStore, DbWorkflowRelayStateStore>();
services.AddWorkflowRelayService();
services.AddFlowBus();

await flowBus.InitiateAsync(new FlowInitiateRequest {
    WorkflowName = "submission-review",
    EntityId = submissionGuid,
    StartEvent = "4100",
    Actor = reviewerEmail
}, ct);
Example

Minimal example

[LifeCycleDefinition("loan-approval")]
public sealed class LoanApprovalRelay : WorkflowRelayBase {
    private readonly ILoanReviewService _review;
    private readonly IAuditService _audit;

    public LoanApprovalRelay(ILoanReviewService review, IAuditService audit) {
        _review = review;
        _audit = audit;
    }

    protected override string DefinitionJson => """
    { "name": "loan-approval",
      "states": [{ "name": "Submitted", "is_initial": true }, { "name": "UnderReview" }, { "name": "Approved", "is_final": true }],
      "transitions": [{ "event": 4000, "from": "Submitted", "to": "UnderReview" }, { "event": 4001, "from": "UnderReview", "to": "Approved" }] }
    """;

    protected override string? PolicyJson => """
    { "policy_name": "loan-approval",
      "rules": [{ "state": "UnderReview", "via": 4000, "complete": { "success": 4001 }, "emit": [{ "route": "APP.LOAN.AUDIT", "type": "effect" }] }] }
    """;

    protected override void ConfigureRelay(WorkflowRelay relay) {
        relay.On(4000, async ctx => await _review.ValidateAsync(ctx.EntityRef));
        relay.OnHook("APP.LOAN.AUDIT", async ctx => { await _audit.WriteAsync(ctx.EntityRef); return true; });
    }
}
var result = await relayHost.InitiateAsync(new FlowInitiateRequest {
    WorkflowName = "loan-approval",
    EntityId = "loan-123",
    StartEvent = "4000",
    Actor = "analyst"
}, ct);
Migration path: if the workflow later needs durable retries, inbox/outbox tracking, or engine-monitored timeouts, keep the same protocol JSON and move the workload to the full engine + consumer runtime.
